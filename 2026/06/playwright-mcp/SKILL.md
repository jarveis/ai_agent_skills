# playwright-mcp

## 什么时候使用

在以下任务中使用这个 skill：

- 需要打开网页确认 UI 是否真的可用。
- 需要复现用户描述的浏览器端 Bug。
- 需要检查按钮、表单、弹窗、导航、响应式布局。
- 需要让 Agent 观察页面状态，而不是只靠源码推断。
- 需要把自然语言测试步骤转成 Playwright 或端到端测试思路。

## 来源

- 仓库：<https://github.com/microsoft/playwright-mcp>
- 原始形式：MCP Server / 浏览器自动化工具
- 本次整理名称：playwright-mcp

## 核心原则

1. **先验证运行结果，再修改代码**  
   浏览器里的真实页面状态优先于代码推断。

2. **在安全环境中操作**  
   优先使用本地、测试环境或 staging。不要在生产系统中执行支付、删除、批量发送等不可逆操作。

3. **把复现步骤写清楚**  
   每次浏览器验证都应记录 URL、视口、操作步骤、观察到的现象和结论。

4. **区分“能看到”和“能交互”**  
   元素可见不等于可点击；按钮存在不等于流程可完成。

5. **不泄露敏感信息**  
   如需登录，使用测试账号；不要把长期密钥、真实 Cookie 或生产凭据写入 prompt。

## Claude Code 接入方式

Playwright MCP 是 MCP server。推荐在 Claude Code 中这样添加：

```bash
claude mcp add-json playwright '{"command":"npx","args":["@playwright/mcp@latest"]}'
```

检查：

```bash
claude mcp list
claude mcp get playwright
```

如果命令参数格式和你的 Claude Code 版本不一致，使用交互式向导更稳妥：

```bash
claude mcp add
```

然后按提示添加一个名为 `playwright` 的 server，命令为：

```bash
npx @playwright/mcp@latest
```

> 仅安装这个 `SKILL.md` 不会自动提供浏览器工具。真正的浏览器能力来自 Playwright MCP server，必须完成 MCP 接入。

## 工作流程

### 1. 明确目标

先向用户确认：

- 要访问的 URL。
- 是否是本地、测试环境或生产环境。
- 需要复现或验证的具体流程。
- 是否允许登录、提交表单或修改数据。

### 2. 打开页面并观察

使用 Playwright MCP 打开目标页面，记录：

- 页面是否加载成功。
- 当前 URL 和标题。
- 关键元素是否可见。
- 是否有明显报错、空白页、加载状态或控制台异常。

### 3. 执行最小交互路径

只执行完成任务所需的最小操作：

- 点击目标按钮。
- 输入测试数据。
- 切换标签、展开菜单或触发表单校验。
- 观察页面状态变化。

### 4. 给出可复现结论

输出时包含：

- 复现步骤。
- 实际结果。
- 预期结果。
- 初步定位方向。
- 如果已修改代码，说明修改点和再次验证结果。

### 5. 需要时补充测试

如果任务涉及前端稳定性，可以建议或生成 Playwright 测试，但不要把“生成测试”当成已经验证通过。测试仍需运行并报告结果。

## 输出要求

每次使用 Playwright MCP 后，至少输出：

```markdown
## 浏览器验证结果

- URL：...
- 环境：本地 / 测试 / 生产
- 操作步骤：
  1. ...
  2. ...
- 实际结果：...
- 预期结果：...
- 结论：通过 / 未通过 / 需要更多信息
- 相关代码或后续建议：...
```

## 使用示例

```text
使用 playwright-mcp 打开 http://localhost:3000/login，
复现“点击登录后按钮一直转圈”的问题，
记录具体停在哪一步，并定位可能相关的组件或请求。
```

```text
使用 playwright-mcp 检查移动端 390px 宽度下产品详情页，
确认“立即购买”按钮是否可见且可点击。
```
